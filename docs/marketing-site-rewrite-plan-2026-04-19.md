# GastroPrime, аудит текущего сайта и план переписывания вместо Tilda

Дата: 2026-04-19
Домен: https://gastroprime.ru
Цель: заменить Tilda на собственный сильный маркетинговый сайт с нормальным SEO, управляемым через код и помощника.

## 1. Что есть сейчас

### Публичные страницы, найденные по sitemap и навигации
- `/` — главная B2B-страница про корпоративное питание
- `/office` — офисы
- `/cosntructions` — стройки (опечатка в URL)
- `/warehouses` — склады
- `/productions` — производства
- `/healthfood` — рационы / правильное питание
- `/government` — госучреждения
- `/page127018316.html` — форма заявки
- `/shop` и `/page132283616.html` — магазин
- `/page132357906.html` — оферта
- `/page132359876.html` — возврат
- `/page132368286.html` — страница благодарности
- `/page132371166.html` — страница отказа

### Дополнительные признаки
- `robots.txt` и `sitemap.xml` есть
- в `robots.txt` sitemap указан через `http://`, а не `https://`
- в `sitemap.xml` тоже `http://` URL и есть URL с опечаткой: `/cosntructions`
- главная страница тяжёлая, HTML примерно 668 KB
- в HTML главной много ссылок на Tilda, по грубой проверке более 300 упоминаний `tilda`
- явной structured data (`application/ld+json`) на главной не найдено
- `hreflang` не найден

## 2. Что на текущем сайте реально хорошо

1. **Есть понятный оффер**: корпоративное питание, Москва и МО, меню под бюджет, отсрочка, бесплатная дегустация.
2. **Есть сегментация по аудиториям**: офисы, стройки, склады, производства, госучреждения.
3. **Есть попытка говорить языком боли клиента**: дисциплина, горячая еда, логистика, смены, мотивация сотрудников.
4. **Есть базовый SEO-каркас**: отдельные страницы под сегменты, title/description хотя бы у части страниц, sitemap, robots.
5. **Есть уже живой продукт**: `app.gastroprime.ru`, и это большое преимущество, потому что сайт можно связать с реальным кабинетом, демо, меню и заявками.

## 3. Что сейчас плохо и почему сайт стоит переписать

## 3.1. Критичные проблемы доверия

### 1) Сайт выглядит как сборка из нескольких разных историй
- B2B-корпоративное питание
- правильное питание на день
- магазин с другой логикой
- шаблонный магазин с текстом про японских поваров
- юридические и thank-you страницы на технических Tilda URL

Пользователь не получает один цельный образ компании. Это бьёт по конверсии.

### 2) В магазине видны следы шаблона
На страницах магазина остался явный шаблонный мусор:
- текст про "японских поваров"
- фраза про демонстрационный контент шаблона
- несвязанный стиль и смысл

Это серьёзно подрывает доверие. Для B2B это особенно плохо.

### 3) Несогласованность контактов и адресов
На разных страницах встречаются разные телефоны и адреса. Для корпоративного клиента это красный флаг.

## 3.2. SEO и структура

### 4) Много технического мусора и слабая управляемость SEO
- sitemap на `http://`
- robots указывает `http://` sitemap
- опечатка в URL `/cosntructions`
- есть и красивые URL, и старые `pageXXXXX.html`
- часть контента дублируется между страницами
- нет явной схемы редиректов и нормальной canonical-стратегии

### 5) Нет полноценной SEO-архитектуры
Сейчас сайт похож не на выстроенный поисковый актив, а на набор лендингов.
Не хватает:
- контентных хабов
- нормальных кейсов
- FAQ-страниц
- страниц по сегментам с уникальной структурой
- статей / экспертного блога
- schema.org разметки
- нормальных внутренних перелинковок

### 6) Слишком тяжёлая и зависимая от Tilda страница
Tilda даёт быстрый старт, но здесь уже стала ограничением:
- тяжёлый HTML
- лишние скрипты и стили
- сложнее контролировать мета, схемы, контентные шаблоны
- неудобно масштабировать сегментные страницы
- неудобно массово редактировать и поддерживать консистентность

## 3.3. Контент и UX

### 7) Контент местами сырой
Есть формулировки, которые выглядят как черновик или быстрая наброска.
Есть повторения, стилистический разброс, местами слабая аргументация.

### 8) Слишком мало сильных доказательств
Для B2B-питания очень важны:
- реальные кейсы с цифрами
- логистика и SLA
- фото производства
- сертификаты и стандарты
- примеры меню
- процесс внедрения
- ответы для закупок, HR, офис-менеджеров, операционных директоров

Сейчас это есть фрагментарно, но не собрано в убедительную систему.

### 9) Нет ощущения "сильного продукта + сильного сервиса"
Сайт продаёт еду, но пока слабо продаёт систему:
- персональный менеджер
- внедрение питания под графики
- учёт смен
- работа с лимитами бюджета
- стабильность поставок
- цифровой кабинет для заявок и отчётности

А это как раз ваша реальная сила.

Ключевая мысль:

Каждая страница сегмента должна быть не копией, а отдельной посадочной под конкретную боль.

Структура сегментной страницы:
1. оффер под сегмент
2. боли и проблемы сегмента
3. решение GastroPrime
4. пример формата питания
5. логистика / графики / доставка
6. кейс похожего клиента

8. CTA на дегустацию или расчёт

## 6. На чём делать новый сайт

## Рекомендуемый вариант
**Next.js marketing site + контент в MDX/JSON/YAML + текущий сервер + nginx + ваш backend для заявок.**

Почему так:
- хороший SEO-контроль
- server-side / static generation там, где нужно
- легко делать чистые URL, sitemap, robots, schema
- удобно править контент через файлы, а значит я смогу менять сайт быстро и предсказуемо
- можно без боли деплоить на ваш текущий сервер

## Альтернатива
**Astro** тоже подойдёт и даст ещё более лёгкий фронт. Если цель почти целиком маркетинговая и минимум интерактива, Astro может быть даже лучше по скорости.

### Моя рекомендация
Если сайт будет:
- с формами,
- с динамическими кейсами,
- с блогом,
- с возможным будущим личным кабинетом/демо-связкой,

то я бы выбрал **Next.js**.

## 7. Как сделать так, чтобы сайтом можно было управлять через меня, а не через Tilda

Нужно хранить контент **не в визуальном конструкторе**, а в понятной структуре проекта.

Пример структуры:
- `content/pages/home.mdx`
- `content/pages/office.mdx`
- `content/pages/warehouses.mdx`
- `content/cases/logistics-hub.mdx`
- `content/faq/*.mdx`
- `content/menu/*.json`
- `config/navigation.ts`
- `config/seo.ts`
- `data/company.ts`

Тогда ты можешь писать мне:
- "обнови телефон на всём сайте"
- "добавь кейс для складов"
- "измени оффер на главной"
- "добавь FAQ про постоплату"
- "сделай новую страницу для заводов"

и я это буду менять напрямую в коде без Tilda.

## 8. Что обязательно сделать по SEO в новой версии

1. Чистые URL
2. 301-редиректы со всех Tilda URL
3. `https` canonical везде
4. корректный `robots.txt`
5. динамический `sitemap.xml`
6. уникальные title / description на каждую страницу
7. правильная структура H1-H2-H3
8. schema.org:
   - Organization
   - LocalBusiness / FoodEstablishment при необходимости
   - BreadcrumbList
   - FAQPage
   - Product / Service там, где уместно
9. OpenGraph / Telegram / WhatsApp превью
10. сжатые изображения, WebP/AVIF
11. Core Web Vitals без Tilda-мусора
12. внутренняя перелинковка между сегментами, кейсами, меню, FAQ
13. отдельные страницы под важные поисковые намерения

## 9. Как переехать с Tilda без потери SEO

### Правильный путь
1. Собрать новый сайт на staging, например `new.gastroprime.ru`
2. Повторить текущие важные страницы в новой версии
3. Заранее составить таблицу редиректов:
   - `/cosntructions` -> `/construction`
   - `/page127018316.html` -> `/request`
   - `/page132357906.html` -> `/offer`
   - `/page132359876.html` -> `/refund`
   - `/page132368286.html` -> `/thanks`
   - `/page132371166.html` -> `/payment-failed`
   - legacy `page982...` -> нормальные сегментные URL
4. Проверить мета, canonical, robots, sitemap
5. Подключить формы к вашему backend
6. Только после этого переключить nginx на новый сайт
7. Оставить 301-редиректы и не ломать старые URL

## 10. Как я бы деплоил это на ваш сервер

### Вариант 1, рекомендованный
- новый проект в репозитории, например `/root/gastroprime/site`
- сборка в production через отдельный скрипт, например `scripts/deploy-site.sh`
- nginx:
  - `gastroprime.ru` -> новый сайт
  - `app.gastroprime.ru` -> текущий кабинет
  - `api.gastroprime.ru` -> backend API

### Если сайт статический
Можно собирать в папку типа `/var/www/gastroprime-site` и nginx будет раздавать статику напрямую.

### Если сайт на Next.js
Можно держать отдельный pm2-процесс, например `gastro-site`, и проксировать nginx на него.
